## 一、上架时间线（T-60 ~ T+7）

为避免医美类应用常见的"临上架才发现缺资质"问题，建议按以下时间线推进：

| 时间节点 | 关键事项 |
|---|---|
| T-60 天 | 公司主体资质、软著（可加急 1500~3000 元 / 1-3 工作日）、**工信部 App 备案**、**ICP 备案**、公安备案、医美合作机构资质收集 |
| T-45 天 | 国内各开发者账号注册（华为/小米/OPPO/vivo/应用宝/荣耀）、申请 App Store Developer（$99/年）、Google Play Console（$25 一次性） |
| T-30 天 | 阿里云短信签名/模板、隐私政策部署至公网 URL、内部真机测试 |
| T-14 天 | iOS 证书/描述文件、Android 签名证书、截图/icon/启动图素材、准备演示账号与短信万能验证码 |
| T-7 天  | 隐私政策/SDK 清单定稿、免责声明位置复查、无网/弱网/拒绝权限路径测试 |
| T-3 天  | 最终真机 QA（iOS/Android 各 2 台不同机型）、Bugly/Sentry 接入、灰度发布计划确认 |
| T-0    | 先提 iOS（审核 1-7 天），再提安卓（利用 iOS 审核期打磨安卓素材） |
| T+1 ~ T+7 | 审核进度跟踪、审核员问题回复、上架后灰度 5% → 20% → 100%、舆情与线索监控 |

> iOS 首发可以申请 Expedited Review（App Review Board → Expedited Request），每年免费 2 次。

### 2.2 医美类专项资质（核心，没有=无法上架）

国内安卓商店对医美类 App 特别严格，以下至少准备一项：

- 《**互联网药品信息服务资格证**》（国家食药总局核发）——若 App 介绍具体药品
- 《**互联网医疗保健信息服务审核同意书**》——若涉及医疗保健信息
- 合作医疗机构的 《**医疗广告审查证明**》（含审查证明文号，登记在每条广告/详情页底部）
- 《**医疗机构执业许可证**》副本——合作机构须为持证机构
- 合作机构授权书（授权你方展示其案例/信息）

> 特别提示：如果 App 仅做"科普信息展示 + 信息留言收集"，不展示具体机构/医师/药品广告，审核通过率会高很多；但留言收集后共享给医疗机构的流程，需在隐私政策中清晰披露。

## 三、应用配置

### 3.1 manifest.json 关键配置（已按合规要求更新）

```
基础配置：
  应用名称：靓人科普
  应用描述：医学美容科普信息平台，提供项目知识、问答、案例信息展示。本应用不提供诊疗服务
  versionName：1.0.0
  versionCode：1（每次发布 +1）
  AppID：HBuilderX 生成的 uni-app AppID（与 iOS Bundle ID 不同）

Android：
  packagename：com.liangren.kepu（一经发布不可修改）
  minSdkVersion：23（Android 6.0，对应权限动态申请模型）
  targetSdkVersion：34（Google Play 2026 要求 ≥35，可升到 35）
  abiFilters：armeabi-v7a + arm64-v8a（华为要求必须含 64 位）

iOS：
  Bundle ID：com.<yourcompany>.liangren（在 Apple Developer 后台创建）
  idfa：false（本应用不使用 IDFA）
```

### 3.2 包名命名规则

- Android 包名：`com.<公司域名反写>.liangren`，例如 `com.yourcompany.liangren`
- iOS Bundle ID 与 Android 包名保持一致
- **一旦在任一商店上线，永远不得修改**

### 3.3 版本号规则

- `versionName`（显示给用户）：语义化版本 `主.次.修订`，如 `1.0.0`、`1.1.0`
- `versionCode`（整数）：每次提交必须严格递增，各商店之间最好共用序号

### 3.4 应用图标 / 启动图

**图标要求：**
- Android：48/72/96/144/192 px（PNG，不透明，不带文字）
- iOS：1024×1024 pt（**无透明 / 无圆角 / 无文字水印**，Apple 会按 4.0 拒审含水印图标）
- 各尺寸在 HBuilderX → manifest.json → App 图标可一键生成

**启动页要求：**
- iOS：使用 LaunchScreen Storyboard 或 1125×2436（iPhone X 机型）等多尺寸图
- 不要使用 DCloud 默认欢迎图（有水印，iOS 会拒审 4.0）
- 建议：白底 + 居中 logo + 底部 slogan，展示时长 1-2s

**应用商店截图（每个商店 3-5 张）：**
- Android：1080×1920 竖屏，必须真机截图
- iOS：5.5 英寸（1242×2208）和 6.5 英寸（1242×2688）各至少 1 张，另 6.7 英寸推荐
- 截图不能含有"免费"、"最佳"、"保证"等承诺性词，也不能伪造效果图

### 3.5 权限最小化（manifest 已配置）

**只申请以下权限：**
- `android.permission.INTERNET`（联网）
- `android.permission.ACCESS_NETWORK_STATE`（网络状态）
- `android.permission.ACCESS_WIFI_STATE`（Wi-Fi 状态）

**已移除的权限：**
- ❌ 剪贴板读取（`READ_CLIPBOARD`）— 合规要求，避免 iOS 系统提示/审核风险
- ❌ 设备标识（`READ_PHONE_STATE`）— 不再收集 IMEI

> 本应用明确不申请：摄像头、麦克风、位置、通讯录、相册、日历、运动数据等敏感权限，降低审核风险。

### 3.6 Android 14 / 15 兼容注意

- **Android 14**（targetSdk 34）强制：前台服务需声明 `foregroundServiceType`（本项目无前台服务，不受影响）
- **Android 14** 禁止安装 targetSdk < 23 的 APK（本项目已 target 34，无影响）
- **Android 15**（targetSdk 35）**Google Play 2026 年 8 月起强制要求**，届时需升级：
  ```
  app-plus.distribute.android.targetSdkVersion: 35
  ```
  升级前需回归：`rich-text` 长文章渲染、Webview、分享、推送等
- 各国内商店（华为 HMS 等）可能晚一年跟进，关注官方公告

### 3.7 iOS 设备标识符政策

- 本项目**不使用** IDFA（`idfa: false`）、IDFV（不主动读取）、`identifierForVendor`
- 设备标识一律使用 **uni-id 返回的 token** 作为用户标识
- 如未来接入 **IDFV**（例如用作设备去重），需在 `PrivacyInfo.xcprivacy` 声明：
  - `NSPrivacyAccessedAPITypes`：`NSPrivacyAccessedAPICategoryUserDefaults`（理由 `CA92.1`）
  - `NSPrivacyAccessedAPICategoryFileTimestamp`（如读文件时间戳，理由 `C617.1`）
- 2024-05 起 Apple 强制校验 Required Reason API，遗漏会被 ITMS-91053 拒审

## 四、合规要求（重点）

### 4.1 隐私政策部署

本应用内置的隐私政策仅作为 App 内可读取版本，**各大应用商店与工信部均要求一个公网 URL**：

1. 从 `pages/webview/index.vue` 提取 `privacyHtml` 和 `agreementHtml`，导出为静态 HTML
2. 部署至已备案域名（推荐 uniCloud 静态网站托管 + 自有备案域名 CNAME）
3. 确保两个 URL 可公开访问：
   - 隐私政策：`https://www.liangren.com/privacy.html`
   - 用户协议：`https://www.liangren.com/agreement.html`
4. 将 URL 同步至：
   - 各安卓商店应用详情的"隐私政策链接"
   - App Store Connect → App 隐私 → 隐私政策 URL
   - `pages/webview/index.vue` 的 `ALLOWED_HOSTS` 白名单

### 4.2 个人信息收集清单（必须写入隐私政策）

| 信息类型 | 收集场景 | 用途 | 是否可拒绝 |
|---|---|---|---|
| 手机号 | 注册/登录 | 身份验证 | 是（可不登录浏览） |
| 姓名、城市、感兴趣内容 | 在线留言 | 共享给合作医疗机构 | 是 |
| 设备型号、OS、App 版本 | App 启动 | 适配与稳定性 | 否（基础能力） |
| IP 地址 | 请求云函数 | 风控 / 频率限制 | 否（基础能力） |
| click_id（广告归因参数） | 从广告链接进入 | 转化回传 | 否（无广告则无此数据） |

> ⚠️ 本应用已移除"剪贴板读取"，请勿在后续迭代中再次引入。若必须引入，请在 iOS 端配合 ATT 框架（参见 6.5），并在隐私政策的 SDK 清单中单独列出。

### 4.3 第三方 SDK 目录（必须写入隐私政策）

| SDK / 服务 | 提供方 | 收集信息 | 用途 |
|---|---|---|---|
| uni-app 运行时 | DCloud | 设备信息、崩溃日志 | 应用框架 |
| uniCloud 云服务 | DCloud / 阿里云 | 请求 IP、调用日志 | 后端服务 |
| uni-id 用户体系 | DCloud | 手机号、登录日志 | 账号注册登录 |
| 阿里云短信服务 | 阿里云 | 手机号 | 发送短信验证码 |
| 巨量引擎转化回传 API | 巨量引擎 | click_id、事件类型、时间戳 | 广告归因（不含手机号） |

### 4.4 权限申请最佳实践（动态权限 + 目的明示）

- 所有敏感权限使用前需二次弹窗说明用途
- 被拒绝后不得反复弹窗骚扰
- 不同意隐私政策时应进入**"仅浏览模式"**（本项目已实现：`privacy_basic_mode`），**不得**通过 `plus.runtime.quit()` 退出 App（iOS 不允许 + 工信部违规）

### 4.5 医美广告合规要点（重点，审核最易翻车）

根据《医疗广告管理办法》《广告法》第十六条：

#### ❌ 全站禁止出现的用语
- "最佳"、"第一"、"最新技术"、"唯一"、"绝对安全"
- "免费"（改为"在线留言"/"了解科普信息"）
- "保证"、"承诺"、"100%"、"彻底根治"
- "专业医师 1 对 1 解答"、"名医亲诊"（未经广告审查不得使用）
- "疗效"、"术后快速恢复"等效果承诺

#### ✅ 推荐用语
- "科普信息"、"相关资料"、"了解更多"
- "留言咨询"、"在线留言"
- "以面诊结果为准"、"请咨询执业医师"

#### ✅ 必须出现的免责声明
- 首页常驻：**"本平台内容仅供科普参考，不构成诊疗建议，就医请咨询执业医师"**（本项目已实现）
- 文章详情底部：**"本文内容仅供科普参考，不构成医疗、诊断或治疗建议..."**（本项目已实现）
- 案例展示顶部：**"以下案例仅供科普参考，个体效果因人而异..."**（本项目已实现）
- 表单提交前：**"提交即授权将信息共享给合作医疗机构..."**（本项目已实现）

### 4.6 账号注销（iOS 5.1.1(v) 强制 + 工信部强制）

- App 内必须能直接完成注销（不得只提示"联系客服"）
- 本项目已接通至 `uni_modules/uni-id-pages/pages/userinfo/deactivate/deactivate`
- 注销后数据处理需在隐私政策中说明时效（本项目政策：15 工作日内删除或匿名化）

### 4.7 未成年人保护

- App 定位 18+ 成年用户
- 首页/设置页提供**青少年模式**入口（本项目已实现于"设置-青少年模式"）
- 隐私政策需有"未成年人保护专章"
- App Store 年龄分级建议 **17+**（医美 + 医疗信息）

### 4.8 冷启动事件时序（合规核心）

**严格遵守**："未同意隐私协议前，不得发起任何网络请求 / 不得收集任何设备信息"。

本项目已实现的正确时序：

```
App 冷启动
   │
   ├─ 展示启动图（本地静态）
   ├─ 进入 pages/index/index.vue
   ├─ 检查 storage 中是否有 privacy_agreed
   │     ├─ 已同意 → 进入正常逻辑
   │     └─ 未同意 → 展示隐私弹窗（仅本地 HTML，不请求云函数）
   │
   └─ 用户点击"同意" 后：
         ├─ 写入 privacy_agreed = true
         ├─ 解析 launch options 的 click_id（不读剪贴板）
         ├─ 上报 app_install 事件（巨量回传）
         ├─ 首次调用 uniCloud 获取内容（get-content）
         └─ 注册推送 / 监听等（如未来接入）
```

**审核员会用抓包工具（Charles）验证这一时序**。若在弹窗前发现任何网络请求，即判违规。

### 4.9 弱网 / 无网 / 拒绝权限下的可用性

iOS / 国内商店均会测试以下路径，每条都不得崩溃：

- ❌ 断网启动：应展示"网络异常"提示，不能白屏
- ❌ 弱网（2G）加载列表：应有 loading 状态和超时兜底
- ❌ 用户拒绝短信验证码（不登录）：App **仍可浏览** 科普内容（仅浏览模式）
- ❌ 用户拒绝所有权限：App **仍可浏览**

本项目已实现：`utils/content-api.js` 云失败时自动 fallback 到本地 mock 数据，保证断网时首页/列表可用。

## 五、Android 打包与上架

### 5.1 生成签名证书

```bash
keytool -genkey -alias liangren -keyalg RSA -keysize 2048 \
  -validity 36500 -keystore liangren.keystore
```

**密码、别名、.keystore 文件必须加密备份（如公司 1Password）。丢失 = 无法发布任何后续版本。**

### 5.2 HBuilderX 云打包

1. 菜单 → 发行 → 原生 App-云打包 → Android
2. 填写签名信息（别名 / 密码 / keystore 文件）
3. 勾选 `armeabi-v7a` + `arm64-v8a`
4. 打包完成下载 APK

### 5.3 加固与隐私合规检测（强烈建议）

- **加固**：360 加固保（免费版）/ 腾讯乐固（免费），医美类尤其建议
- **隐私检测**：各商店上传前自测，推荐
  - 华为 Piracy Check
  - 小米隐私合规检测工具
  - 360 应用隐私合规检测
  - 国家互联网应急中心"App 隐私合规检测系统"

### 5.4 各应用商店提交

**华为应用市场（AppGallery）**
1. 我的应用 → 新建应用 → 分类选"医疗健康"或"实用工具"
2. 上传 APK、截图（≥3 张）、隐私政策 URL、软著证书
3. 医美类会额外要求《医疗广告审查证明》或合作机构资质
4. 审核周期 1-3 工作日

**小米 / OPPO / vivo**
流程类似，注意：
- 小米要求《软件功能说明》word 文档
- OPPO / vivo 医美类需额外提交资质与广告审查说明

**应用宝（腾讯）**
- 需通过微信开放平台关联（即使不用微信登录，也需建立关联）
- 应用宝对医美类"**案例展示**"特别敏感，建议弱化案例图片、强化科普文字

**荣耀 / 魅族 / 三星**
基本同华为流程，荣耀已独立于华为账号。

### 5.5 Android 审核常见被拒原因与对策

确保以下集合存在（见 `uniCloud-aliyun/database/*.schema.json`）：

- `leads` — 留言线索
- `feedbacks` — 用户反馈
- `clients` — 合作医疗机构

- `sms-logs` — 短信发送日志
- `articles` — 科普文章（内容来源于 `utils/mock-data.js`，已洗白）
- `qa` — 问答
- `tips` — 美丽贴士
- `uni-id-users` — 用户表（uni-id 自动创建）

#### 7.4.1 种子数据导入（首次部署）

> 首版所有内容来自本地洗白后的 mock 数据，需要一次性灌库。

1. **本地执行**
   ```bash
   node tools/clean-mock-data.js   # 幂等清洗 mock-data.js（保留规则见文件头部注释）
   node tools/gen-seed-data.js     # 生成 uniCloud-aliyun/cloudfunctions/import-seeds/seed-data.generated.js
   ```
2. **上传 `articles`、`qa`、`tips` schema** 到 uniCloud 控制台
3. **上传 `import-seeds` 云函数**，并在其"环境变量"中配置
   - `SEED_IMPORT_SECRET` = 任意强随机字符串（仅运维知道）
4. **在控制台"云函数 → import-seeds → 运行"里传参**：
   ```json
   { "secret": "<SEED_IMPORT_SECRET>" }
   ```
5. 返回 `summary` 显示 `articles.created = 34 / qa.created = 22 / tips.created = 8` 即成功
6. 每次 mock 内容变更后，重新执行 1-4 步可**按 `seed_id` 幂等更新**，不会重复导入

> ⚠️ 首版上架前建议：**导入后进入控制台把人工审核过的文章 `status` 置 1、其余置 0**，
> 避免审核员看到"还未人工核校"的科普文被挂一个内容合规 issue。

#### 7.4.2 运行时数据链路

```
客户端 → uniCloud.callFunction('get-content') → articles/qa/tips
                              │
                              └─ 失败/超时（6s） → utils/content-api.js 本地 fallback 到 mock-data.js
```

- 本地 fallback 仅为开发期 / 云函数失联时的兜底，**上架版本必须确保云函数可用，否则数据可能回退到 mock**
- 首版审核：可以在 uniCloud 控制台把 `get-content` 函数 "开启公网访问"（客户端调用）

### 7.5 数据安全建议

- `leads.phone / leads.name` 建议字段级加密存储（可用 uniCloud 数据库加密字段或自行 AES）
- `leads.submit_ip` 用于风控，保留期建议 ≤6 个月
- 数据库按天自动备份（uniCloud 控制台支持）
- 定时触发 `clean-expired` 云函数清理过期的 click_id / 日志

### 7.6 企业微信 Webhook 安全

- Webhook URL 泄露会被恶意发送消息，务必：
  - 仅通过云函数环境变量传入，不进入前端代码
  - 在群机器人后台设置"可信 IP 列表"（uniCloud 提供的网关 IP）

### 7.7 云函数日志 / 监控 / 告警

**日志**：
- uniCloud 控制台 → 云函数 → 选中函数 → 日志 可以按时间 / 关键字查询
- 默认保留 7 天；涉及留言、广告回传的关键日志建议存库（本项目 `conversion-logs` 已实现）

**监控**：
- 控制台 → 云函数 → 监控 → 查看调用次数 / 平均耗时 / 失败率
- 推荐关注指标：
  - `submit-lead` 失败率 > 3% → 说明风控/验证码异常
  - `report-conversion` 失败率 > 10% → 说明 OE Token 失效
  - `get-content` 平均耗时 > 1s → 说明数据库索引缺失或数据爆涨

**告警**：
- uniCloud 控制台 → 告警策略 → 新建
- 对上述指标设置阈值，触发后通过企微 webhook 发消息到运营群

### 7.8 数据库备份与恢复

- uniCloud 控制台 → 数据库 → 备份策略 → **每日自动备份**（默认保留 7 天，可付费升级到 30 天）
- **手动备份**（重大变更前）：控制台 → 备份与恢复 → 立即备份
- 恢复流程：
  - 联系 uniCloud 工单（提供 spaceId + 备份时间点）
  - 半天内完成（工作日）
- 建议每月一次自主导出 `leads / uni-id-users` 到公司 OSS（非阿里云 uniCloud 空间），做异地多活备份

### 7.9 云函数灰度发布

- 控制台 → 云函数 → 版本管理 → 可以保留历史版本，回滚到任意历史版本
- 重大变更（如改数据库 schema 的函数）发布时：
  - 先上传"开发版"（标识 DEV）
  - 真机测试
  - 替换"默认版"（生产流量）
  - 若发现问题，30 秒内回滚

## 八、巨量引擎回传配置

### 8.1 整体流程

```
用户点击抖音广告
    ↓
进入应用商店下载页（URL Scheme / Universal Link 携带 click_id）
    ↓
用户安装 App 并首次启动
    ↓
App 从启动参数中读取 click_id（不再从剪贴板读取）
    ↓
用户同意隐私协议后 → 回传"激活"事件（含 app_type）
    ↓
用户提交留言表单 → 回传"表单提交"事件
    ↓
新用户注册成功 → 回传"注册"事件（仅首次）
```

### 8.2 获取 Access Token

1. 登录巨量引擎后台（ad.oceanengine.com）
2. 工具 → 转化追踪 → 查看 Access Token（长期 token）
3. 配置到 `OE_ACCESS_TOKEN` 环境变量

### 8.3 定时重试配置

在 uniCloud 控制台 → 云函数 → `retry-conversion` → 定时触发：

```
cron: 0 */5 * * * *    # 每 5 分钟执行一次
```

重试节奏（失败后）：1min → 5min → 15min → 30min → 2h，最多 5 次。

### 8.4 回传事件对照表

| 事件 | OE event_type | 触发时机 | 代码位置 |
|---|---|---|---|
| app_install | 4（激活） | 用户同意隐私协议后首次 | App.vue / pages/index/index.vue |
| form_submit | 6（表单提交） | 提交留言成功 | submit-lead 云函数 |
| register | 2（注册） | 新用户首次注册成功 | utils/login-hook.js |

### 8.5 app_type 字段（重要）

巨量引擎要求 App 端事件携带 `app_type`：
- `1` = Android
- `0` = iOS

本项目 `report-conversion` 云函数已按 `event.app_type` 携带该字段，前端 `reportAppInstall` / `onLoginSuccess` 会自动传入。

### 8.6 click_id 有效期

巨量引擎 click_id 有效期为 24 小时，超期回传会被判定无效。本项目 `getLocalClickId` 已自动清理超期数据。

### 8.7 回传一致性监控

- 上线后每日对比：
  - 巨量引擎后台"数据报表"中的"激活 / 表单提交"数量
  - uniCloud `conversion-logs` 中 `success: true` 的数量
- 差值 > 5% 时需检查：
  - click_id 是否丢失（URL Scheme 跳转中断？）
  - Access Token 是否过期
  - 事件触发时机是否有延迟

## 十二、账号/主体迁移

### 12.1 触发场景

满足以下任一条件时，需要按[《迁移指南》](./迁移指南.md)执行主体变更：

- 项目原由**个人账号**申请，需变更为**公司主体**（医美类商店强制要求）
- 公司工商变更（更名、合并、分立）
- 运营团队剥离至新公司
- 旧账号触发平台"严重违规"被封，应急切换新账号

### 12.2 必读：迁移与上架的关系

- 首次上架建议**直接用公司主体**。本指南所有章节默认假设你在使用公司主体。
- 若已经个人上架并运营一段时间，再转公司：按[《迁移指南》](./迁移指南.md)执行，整体周期约 **30 个工作日**（主要卡在 ICP 重备案、Apple 账号、各安卓商店重新上架）。
- 迁移期间建议**冻结功能迭代**，只接受修 bug，以避免新老环境行为不一致。

### 12.3 迁移导致上架策略的差异

| 项 | 首次上架 | 迁移上架 |
|---|---|---|
| iOS 审核 Review Notes | "首个版本" | 需主动说明"本应用原由个人账号运营，现转至公司主体" |
| 应用商店描述 | 正常 | 禁止写"账号迁移""公司变更"等，以免触发人工审核疑问 |
| 用户公告 | 无 | 应用内 + 微信 + 官网三端公告 |
| 评论保留 | 不涉及 | Google 可保留（App Transfer），其他商店普遍清零 |
| 广告归因 | 从 0 开始 | 老广告主数据**不可迁移**，归档后新广告主从 0 开始 |
